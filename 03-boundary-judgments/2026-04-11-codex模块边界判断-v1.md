# Codex 模块边界判断 v1

## 目标

先把目前已经比较稳定的边界判断收在一页，避免后续源码拆解又回到“所有模块看起来都在做一切”的混沌状态。

## 当前边界判断

### 分发层
- `codex-cli/`：npm / node launcher / 平台二进制选择；不是 runtime 主体

### 入口层
- `codex-rs/cli`：统一命令入口与模式分流层
- `codex-rs/tui`：interactive front-end
- `codex-rs/exec`：non-interactive front-end

### 协议与服务层
- `app-server`：控制平面 / RPC 服务层
- `app-server-client`：调用方 facade
- `app-server-protocol`：共享契约层
- `mcp-server`：Codex 作为 MCP server 的暴露面

### runtime 核心层
- `core`：agent runtime 聚合核心
- `ThreadManager`：全局线程与共享服务协调者
- `CodexThread`：线程级 API 面
- `Codex`：更底层 session / turn loop 容器

### 能力接入层
- `rmcp-client`：外部 MCP transport/client adapter
- `codex-mcp`：MCP runtime integration
- `hooks`：生命周期事件 hook 机制
- `skills`：skill 资产与 runtime skill manager

### 执行与安全层
- `sandboxing`：执行环境边界
- `execpolicy`：策略规则评估
- `approval`：运行时动作门禁

### 持久化层
- `rollout`：durable history truth source（当前仍需继续确认具体 crate 细节）
- `state`：SQLite metadata/log mirror
- `config`：config schema / layer / provenance

### 嵌入层
- TypeScript SDK：CLI embedding 路线
- Python SDK：app-server embedding 路线

## 当前最重要的非边界判断

### 1. 不要把 TUI 当主体
TUI 是正体验，但不是系统本体。

### 2. 不要把 `codex-cli/` 当主体
npm 安装路径很显眼，但它只是 distribution shell。

### 3. `core` 是最像“大脑”的地方
后续深入阅读必须把重心放到 `core` 和它的周边协作层上。

## 仍待进一步确认的边界

1. `core` ↔ `exec` 的更细分工
2. `app-server` ↔ `mcp-server` 的产品边界
3. `rollout` ↔ `state` 的物理/逻辑边界
4. TUI ↔ app-server 的事件流细边界
