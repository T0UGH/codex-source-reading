# Codex 源码导读手册 v2｜对外介绍文案 v1

## 一句话介绍

**《Codex 源码导读手册 v2》是一套按 6 个认知台阶重组 OpenAI Codex 内部结构的中文源码导读手册。**

它不是带你平铺直叙地逛源码目录，而是把 Codex 拆成一条连续问题链：先看清它到底是什么系统，再看它怎样跑起来、怎样持续存在、怎样暴露成控制面、怎样把动作落成执行会话，最后怎样长出审查、协作与更高层 runtime 组织。

---

## 短版介绍（适合 GitHub / 飞书 / 社媒简介）

如果你想真正读懂 OpenAI Codex，不够只看几个模块名，也不够只盯着某个函数。更难的是：**你得先建立整机视角，再沿着 runtime、恢复、控制面、执行会话和高层组织结构逐层往里走。**

《Codex 源码导读手册 v2》就是按这条认知递进重组出来的一套中文源码导读手册。

这套手册目前按 6 卷组织：

1. 系统入口与总图
2. runtime core 主线
3. 状态、恢复与持续工作线
4. 控制面与 app-server 语义
5. 统一执行子系统
6. 审查协作与高级 runtime

适合：

- 不满足于“功能介绍”，而想真正建立系统级理解的读者

## 长版介绍（适合项目页 / 介绍文档）

### 这套手册在做什么

市面上很多源码导读，往往会掉进两种常见问题：

1. **材料导向**：按目录、按模块、按文件平铺展开，读到最后还是不知道整台系统怎么连起来
2. **局部导向**：只盯住某条调用链或某个热点模块，看完局部更懂了，但整体反而更散

Codex 这种系统尤其容易让人误读。

因为你很容易把它看成：
- 一个 npm CLI
- 一个终端里的 TUI
- 一个很厚的 app-server
- 一组散落的 agent feature

但如果只停在这些表面角色，后面再读 runtime core、恢复路径、控制面、统一执行、review / guardian / collab / memories，就会一直像在看零散部件，而不是同一台机器。

这套手册要解决的，就是这个问题。

### 它为什么不是普通源码笔记

这版 guidebook v2 的核心原则不是“归档材料”，而是“建立认知递进”。

整本书围绕一条连续问题链展开：

- Codex 到底是什么系统
- 一条请求怎么真正进入 runtime 主线
- 这条工作线为什么不会中断后就散掉
- runtime 怎样被暴露成稳定控制面
- 动作怎样被装成可管理执行会话
- 这些能力又怎样被组织成更高层 runtime

也就是说，这不是一本“模块说明书”，而是一套**系统理解手册**。

### 这套手册适合谁

这套手册最适合下面几类读者：

- 想系统理解 OpenAI Codex 内部结构的服务端 / 基础设施工程师
- 关注 agent runtime、执行控制、状态恢复、多主体协作的人
- 想从具体实现里反推 agent 系统设计方法的人
- 不满足于“知道它能做什么”，而想理解“它为什么这样组织”的读者

### 当前结构

当前公开入口已经切到 guidebook v2，按 6 卷组织：

- **卷一｜系统入口与总图**：先立整机总图与系统前门
- **卷二｜runtime core 主线**：解释一轮工作回合怎样跑起来
- **卷三｜状态、恢复与持续工作线**：解释系统为什么能继续存在
- **卷四｜控制面与 app-server 语义**：解释 runtime 怎样被暴露成稳定控制面
- **卷五｜统一执行子系统**：解释动作怎样变成执行会话
- **卷六｜审查协作与高级 runtime**：解释系统怎样继续长成更高层运行组织

### 现在这套手册最值得看的地方

目前这套手册已经不只是有正文，还开始形成：

- 卷级问题链
- 卷级入口
- 整本书总入口
- 一批高价值解释图
- 整书认知递进总图
- 公开站点阅读入口

所以它现在更像一本逐步成型的技术手册，而不只是一个源码阅读仓库。
