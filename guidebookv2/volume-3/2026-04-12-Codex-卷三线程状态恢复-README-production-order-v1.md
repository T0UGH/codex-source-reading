# Codex 卷三执行说明

> 本文件不是讨论稿，也不是灵感备忘。  
> **本文件是 Codex 新卷三的执行协议。**  
> 卷三生产必须按本文件推进。

---

## 一、总则

### 1. 卷三必须按本执行说明推进

卷三必须按本文件规定的：

- 文章顺序
- 依赖关系
- 串并行边界
- 验收与回退规则

执行。

**不得临时改卷三主问题，不得写着写着又退回“数据库恢复专题”或“turn 名词表”。**

如果卷三结构判断发生变化，必须先修改：

1. 卷三 spec / 卷级 index
2. 卷三写作卡片
3. 本执行说明

三者一致后，才允许继续推进卷三。

### 2. 卷三当前唯一有效主问题

卷三唯一主问题是：

> **Codex 为什么不是“一次命令执行就结束”，而是一个可以持续、恢复、继续运行的线程化系统。**

因此卷三写作期间：

- **不得把卷三降成 state / SQLite 文件讲解**
- **不得把卷三写成 replay API 说明书**
- **不得把卷三偷换成 app-server 控制面卷**
- **不得把卷三提前吃掉 unified-exec 执行子系统职责**

### 3. 卷三的唯一收口不得改变

卷三最后必须收束为：

> **Codex 的持续工作能力，不是“存了更多历史”，而是 runtime 能重建一条还能继续跑的工作线。**

如果最后一篇没有把这层判断收住，卷三就不算真正完成。

### 4. 卷三当前唯一有效结构

卷三当前唯一有效结构为 5 篇：

1. 为什么 rollout 才是正文真相源，而 SQLite 不是恢复核心
2. 为什么恢复更像 replay，而不是“查库拼对象”
3. thread、turn 与 pending request 是怎么一起组成持续工作线的
4. 为什么 turn-history 不是 event log 镜像，而是一层 semantic projection
5. `active_turn_snapshot`、`handle_user_message` 与当前工作视图的边界

对象主线顺序固定为：

> **真相源分层 → replay 恢复心智 → 持续工作单元 → 语义投影层 → 当前工作视图收口**

### 5. 本文件优先于临场发挥

卷三执行期间：

- 不允许“先挑最顺手的一篇写”
- 不允许“先把 turn-history 那篇写爽再说”
- 不允许“边写边把卷三控制面带进来”

**临场发挥不能覆盖本文件。**

## 二、卷三文章角色分工（固定）

### 01｜真相源澄清篇
- 负责把 rollout / SQLite / sidecar 的职责切开
- 负责打掉“有数据库 = 数据库负责恢复”的直觉误读
- 它是卷三入口，不是细节展开篇

### 02｜恢复心智锚点篇
- 负责把 recovery 从“查库重建”切成“replay / reconstruction”
- 它是卷三真正的机制锚点篇
- 后面 03 / 04 / 05 都默认建立在这篇判断之上

### 03｜持续工作单元篇
- 负责把 thread / turn / pending request 组织成 runtime 工作线
- 它不是 turn-history 篇，也不是控制面篇
- 作用是把“恢复”推进到“恢复回来之后，到底是什么东西在继续工作”

### 04｜语义投影展开篇
- 负责把 turn-history 立成 semantic projection
- 它不能回头重写 02，也不能抢 05 的“当前工作视图”职责

### 05｜卷尾收口篇
- 负责把 active turn / user message / outward snapshot 收成当前工作视图边界
- 负责把卷三压回“continuable runtime working line”
- 它必须最后写

## 三、卷三唯一允许的运行策略

卷三必须采用：

> **前半严格串行，后半有限串行展开，卷尾强制收口。**

解释如下：

1. **01 和 02 必须严格串行**
2. **03 只能在 02 通过后启动**
3. **04 只能在 03 通过后启动**
4. **05 必须最后启动**

卷三**不得**采用：

- 5 篇一起全并行
- 从 04 turn-history 直接起跑
- 先写 05 再倒推前文
- 在 03 里提前讲 app-server 控制面
- 在 04 / 05 里重新解释 rollout / SQLite 分层

## 四、分回合强制执行顺序

## 回合 1：先立卷三底座

### 本回合包含文章
1. 为什么 rollout 才是正文真相源，而 SQLite 不是恢复核心
2. 为什么恢复更像 replay，而不是“查库拼对象”

### 强制顺序
**01 → 02**

### 执行规则
- **必须严格串行**
- **01 未通过，不得启动 02**
- **02 必须建立为卷三机制锚点，不得写成 01 的重复展开**

### 为什么必须这样跑
因为卷三最容易发生的误判，不是后半段边界没切清，而是前两篇没把读者心智扳过来：

- 如果 01 没立住，读者会继续以为 SQLite 才是恢复主脑
- 如果 02 没立住，后面所有篇都会滑回“查库拼对象”心智

所以回合 1 必须先完成：

> **真相源分层 → replay 恢复心智**

### 回合 1 验收目标
主会话必须确认：

- rollout / SQLite 的职责已经切开
- recovery 已被立成 replay / reconstruction，而不是 ORM 式查表
- 01 和 02 没有互相重复
- 卷三的“持续存在”问题已经真正起势

未满足以上条件：

> **不得进入回合 2。**

---

## 回合 2：把恢复推进成工作线

### 本回合包含文章
3. thread、turn 与 pending request 是怎么一起组成持续工作线的
4. 为什么 turn-history 不是 event log 镜像，而是一层 semantic projection

### 强制顺序
**03 → 04**

### 执行规则
- **03 未通过，不得启动 04**
- **03 必须先把 thread / turn / pending request 立成持续工作单元**
- **04 不得越界重写 03 的对象组织职责**
- **04 只能在 03 稳定后，进一步解释 turn-history 的语义投影性质**

### 为什么必须这样跑
因为 03 和 04 看起来都在讲 turn，但它们回答的不是同一个问题：

- **03 回答：恢复回来之后，什么东西在继续工作**
- **04 回答：这些工作痕迹为什么不是原始 event 的机械镜像**

如果两篇并行或倒序，最容易出现：

- 04 抢写 03 的持续工作单元职责
- 03 过早陷入 history builder 细节
- 卷三从“runtime 持续条件”滑成“turn-history 专题”

所以回合 2 必须按：

> **工作线对象成立 → 语义投影解释**

### 回合 2 验收目标
主会话必须确认：

- 03 已把 thread / turn / pending request 写成一条持续工作线
- 04 已把 turn-history 写成 semantic projection，而不是 event dump
- 03 / 04 的问题边界已经清楚分开
- 卷三没有漂移成 turn-history 单专题

未满足以上条件：

> **不得进入回合 3。**

---

## 回合 3：卷尾强制收口

### 本回合包含文章
5. `active_turn_snapshot`、`handle_user_message` 与当前工作视图的边界

### 强制顺序
**05 最后单独启动**

### 执行规则
- **05 必须在 01-04 全部通过后启动**
- **05 不得重讲 rollout / SQLite / replay 基础分层**
- **05 不得提前进入卷三的控制面 facade / request semantics**
- **05 必须把卷三收束到当前工作视图与 continuable working line**

### 为什么必须最后写
因为 05 不是另一个局部机制篇，而是卷三总收口篇：

- 它要把前面的 persistence / replay / thread-turn / semantic projection 都压回“当前还能继续工作的视图边界”
- 它要解释 `active_turn_snapshot` 与 `handle_user_message` 为什么不是普通 getter / append
- 它要把卷三顺利导到卷三：已经成立的 runtime 工作线，接下来怎样被投影成控制面

如果 05 提前写，最容易出现：

- 提前吃掉 03 / 04 的职责
- 把卷三写成一些局部边界函数说明
- 收口还没准备好，就先做总结

### 回合 3 验收目标
主会话必须确认：

- 05 已经把“当前工作视图”立成卷尾判断
- `active_turn_snapshot` / `handle_user_message` 已被解释为边界函数，而不是工具函数
- 卷三最后收束到“runtime 能继续跑的工作线”
- 对卷三的导流是：控制面怎样投影这条已成立的工作线，而不是另起一个 app-server 专题

未满足以上条件：

> **卷三不得宣布完成。**

## 六、当前最重要的执行结论

如果只压一句话，卷三当前唯一允许的生产顺序是：

> **先立真相源分层，再立 replay 心智，再把恢复推进成持续工作线，再解释 semantic projection，最后收口到当前工作视图。**
