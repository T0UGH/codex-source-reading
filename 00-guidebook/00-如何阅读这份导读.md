# 如何阅读这份 Codex 导读

## 先给结论

这份导读**不是 API 文档**，也**不是按目录展开的源码百科**。

它要做的事只有一件：

> **帮你先建立 Codex 的系统总图、主链路和边界判断，再带你进入关键源码。**

如果你直接从仓库目录开始一层层翻，很容易得到很多局部信息，
但很难回答这几个真正重要的问题：

- Codex 的 runtime 主体到底在哪
- `codex-cli/`、`codex-rs/cli`、TUI、`core`、`app-server`、`mcp-server` 分别是什么
- thread / turn / rollout / unified-exec 这些关键对象是怎么串起来的
- 为什么这套系统的正确性很多来自一批小型 reducer / projection / reconciliation helper，而不是一个中心大状态机

所以这份导读的阅读顺序不是“从目录开始”，而是：

1. 先定系统边界
2. 再读主链路
3. 最后再下钻到关键函数

---

## 这份导读适合谁

我默认它服务两类读者。

### 1. 想快速建立系统认知的人
这类读者不想先陷进实现细节，
更关心：
- 系统怎么分层
- 核心运行面在哪
- 主要子系统怎么协作

### 2. 想继续深读源码的人
这类读者更关心：
- 关键文件在哪里
- 哪些函数是 pivot
- 哪些点容易误读
- 后面该沿哪条线继续下钻

如果你属于第一类，先抓总图。
如果你属于第二类，先用总图校准视角，再下钻具体函数。

## 这份材料的三层结构

现在这套仓库材料，已经可以稳定分成三层。

## 1. 导航层

代表文件：
- `index.md`

作用：
- 告诉你先看什么
- 告诉你按哪条线看
- 把全仓材料组织成可读路径

你可以把它理解成：
- **目录入口**
- 但不是正文本身

---

## 2. 正文层

目录：
- `00-guidebook/`

作用：
- 形成一套连续可读的 Codex 系统导读
- 从系统分层、主链路、关键子系统的角度组织内容

这层不是研究原始记录，
而是：
- 已经被重组过的叙事层

你真正要连续阅读的内容，应该主要在这里。

---

## 3. 证据层

目录：
- `01-source-notes/`

作用：
- 保留源码拆解时的判断依据
- 保留关键模块和关键函数的局部研究材料
- 给后续继续深挖的人提供证据入口

这层很重要，
但它不应该替代正文。

更准确地说：

> `01-source-notes/` 是证据仓，不是第一次阅读的主线仓。

## 正确的阅读方式

错误读法通常有两种。

### 错误读法 A：从仓库根目录开始平扫
问题：
- 你会很快迷失在目录和文件名里
- 看到很多局部实现，但不知道它们在系统里的位置

### 错误读法 B：从函数级笔记直接开始
问题：
- 你会理解某个函数很重要
- 但不知道它为什么重要、它服务哪条主线

所以更合理的读法是：

### Step 1：先读总判断
建议先看：
- `03-boundary-judgments/...`
- `STATUS.md`
- `index.md`

目标：
- 先建立整体边界感

### Step 2：选一条主线读正文
建议先选其中一条：
- 系统总图与分层
- app-server / thread / turn 主线
- unified-exec 主线

目标：
- 先抓住一个完整系统面

### Step 3：再下钻关键函数
当你已经知道某条主线怎么跑时，
再去看：
- `ThreadHistoryBuilder::handle_event(...)`
- `active_turn_snapshot(...)`
- `process_chunk(...)`
- `refresh_process_state(...)`
- 这类函数级材料

这时候函数级材料才真正有价值，
因为你能把它放回主线里理解。

## 阅读这份导读前，先记住 6 个核心术语

## 1. thread
可以把它理解成一次会话/任务运行的承载单元。
app-server 对外暴露的很多协议视图，最终都挂在 thread 上。

## 2. turn
thread 里的一个交互轮次。
但它不是简单“用户一句、助手一句”的文本轮次，
而是一个更强语义的单元：会包含 user input、tool 调用、reasoning、状态封口、错误等。

## 3. rollout
可持久化的历史正文真相源。
很多恢复工作最终不是“查数据库拼对象”，而是从 rollout replay 还原出来的。

## 4. app-server
面向客户端 / SDK / TUI 的控制面 facade。
它不是真正的 runtime owner，
但负责把 runtime 的事件和状态投影成稳定协议面。

## 5. unified-exec
不是简单执行 shell 命令的工具函数，
而是一个会话化、可恢复、可交互、可协议化的执行子系统。

## 6. plugin
更像上层 capability packaging unit。
skills、MCP、apps、connectors 的很多关系，要通过 plugin 这层去理解才不会乱。

## 读完整套导读后，你应该得到什么

如果这套导读写得合格，
读完后你至少应该能明确这些判断：

1. `codex-cli/` 只是分发壳，不是主体
2. `codex-rs/cli` 是统一入口层
3. `core` 是 runtime 聚合中心
4. `ThreadManager` 比 app-server 更接近 runtime owner
5. `app-server` 是控制面 facade，不是另一套 runtime
6. turn-history 是 client-facing semantic projection，不是 event log 的原样镜像
7. unified-exec 是 sessionful execution subsystem，不是“多包一层 exec”
8. Codex 的很多一致性来自很多小型边界函数，而不是一个中心大状态机

如果你读完还不能稳定回答这些问题，
说明这套导读还没写到位。

## 推荐阅读路径

### 路径 A：只想先建立全局认知
按这个顺序：
1. 系统总图与分层
2. 状态、持久化与恢复
3. app-server / thread / turn 主线
4. unified-exec 执行子系统

适合：
- 第一次接触 Codex 源码
- 想先搭整体认知框架

### 路径 B：只关心 app-server 和 turn 语义
按这个顺序：
1. app-server / listener / thread 主线
2. turn-history 语义层
3. 相关函数级附录

适合：
- 想搞清恢复、resume、rejoin、turn 视图

### 路径 C：只关心执行链
按这个顺序：
1. unified-exec 执行子系统
2. exec-server / sandbox / environment 相关材料
3. 相关函数级附录

适合：
- 想搞清执行、输出、终态、process store

## 这份导读不会做什么

为了避免误解，也先说清楚它**不会**做什么。

### 1. 不会逐目录罗列所有源码文件
那会把文档写回百科全书。

### 2. 不会把所有函数都展开解释
只会抓关键边界函数。

### 3. 不会把未验证猜想伪装成结论
不确定的内容会留在开放问题里。

### 4. 不会把调研过程流水账当正文
研究过程有价值，但正文更重要的是系统本身。

---

## 如何使用正文和附录

我建议这样用：

- **第一次阅读**：主要读正文，不要深陷附录
- **第二次回查**：把附录当作关键函数索引来用
- **继续深读源码**：从正文小节对应的关键文件 / 关键函数继续下钻

一句话说：

> 正文负责让你理解系统，附录负责让你验证系统。

---

## 小结

这份 Codex 导读最重要的前提不是“信息够多”，
而是“阅读顺序正确”。

正确顺序应该是：
- 先定边界
- 再抓主线
- 最后再看函数级证据

如果你已经准备好了，下一章最应该读的是：

- [01-系统总图与分层.md](01-系统总图与分层.md)

它会先回答整个仓库里最基础、也最容易读歪的一个问题：

> **Codex 到底是一套怎样分层的系统？**
