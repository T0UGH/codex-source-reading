# 代码路径索引：入口与分发

## 这张索引卡解决什么问题

回答两个问题：

1. 用户安装的 `codex` 到底先跑哪层？
2. 真正的主体实现在哪？

## 分发入口

### 顶层 README
- `README.md`
- 结论：公开安装入口主推 `npm i -g @openai/codex` / `brew install --cask codex`

### npm 分发壳
- `codex-cli/package.json`
- `codex-cli/bin/codex.js`
- 结论：npm 包暴露的是 `bin/codex.js`，它负责按平台选择对应原生二进制包并 `spawn()` 真正的 `codex`

## 真正运行入口

### Rust CLI 总入口
- `codex-rs/cli/src/main.rs`
- 结论：真正的统一命令入口在这里；默认无子命令走 interactive TUI

### Rust CLI 支撑层
- `codex-rs/cli/src/lib.rs`
- 结论：这里提供平台相关 sandbox / login 等 CLI 辅助能力，不是完整业务核心

## 主体核心

### 核心库
- `codex-rs/core/src/lib.rs`
- 结论：`codex-core` 是主要 runtime 聚合层，汇总 config、thread、mcp、skills、sandboxing、rollout、tools 等核心能力

### TUI 层
- `codex-rs/tui/src/lib.rs`
- 结论：`codex-tui` 是交互界面层，依赖大量 core / app-server / protocol 相关能力

### 配置 / 状态层
- `codex-rs/config/src/lib.rs`
- `codex-rs/state/src/lib.rs`
- 结论：配置建模与 SQLite 状态分别单独成 crate

## 当前边界判断

- `codex-cli/` = 分发壳，不是主体
- `codex-rs/cli` = 命令入口，不是全部核心
- `codex-rs/core` = runtime 聚合核心
- `codex-rs/tui` = 交互界面层
- `sdk/` = 对外 embedding 层，不是主体 runtime
