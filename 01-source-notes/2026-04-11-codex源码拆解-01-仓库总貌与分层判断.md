# Codex 源码拆解 01：仓库总貌与分层判断

## 这篇看什么

先回答最基础的问题：**Codex 到底是什么形态的系统？**

## 先给结论

Codex 不是一个“单一 CLI 小工具”，而是一个 **Rust 核心 runtime 平台**。

当前仓库至少可以分成 4 层：

1. **分发层**：`codex-cli/`、release、brew/npm 安装路径
2. **界面/入口层**：`codex-rs/cli`、`codex-rs/tui`、app-server、desktop app
3. **runtime 核心层**：`codex-rs/core`
4. **底座能力层**：config、state、sandboxing、MCP、skills、protocol、SDK 等

### 关键证据
- `README.md`：公开对外主定位是 “Codex CLI is a coding agent from OpenAI that runs locally on your computer.”
- `codex-rs/Cargo.toml`：workspace 很大，说明实现不是单 crate，而是拆成大量职责模块
- `codex-rs/core/src/lib.rs`：导出面极宽，说明 `codex-core` 是 runtime 聚合核心
- `codex-rs/tui/src/lib.rs`：TUI 明显是上层 UI/interaction layer，不是唯一入口

## 初步结构判断

### 1. `codex-cli/` 不是实现主体
它主要负责安装和启动原生二进制。

### 2. `codex-rs/` 才是真正主战场
而且不是单块代码，而是 workspace + 多 crate 协作。

### 3. `codex-core` 是最像“大脑”的地方
从导出面看，线程、配置、MCP、skills、rollout、sandboxing、tools、state bridge 都往这里汇聚。

### 4. `codex-tui` 是重要入口，但不是系统全部
它更像一个“交互前端”，并且已经明显和 app-server / protocol / core 协同。

### 5. Codex 的定位更接近“本地 agent runtime”
而不是传统命令行工具。

## 当前边界判断

- `codex-cli/`：分发壳
- `codex-rs/cli`：命令入口聚合
- `codex-rs/tui`：交互层
- `codex-rs/core`：runtime 聚合核心
- `sdk/`：嵌入式调用层
- `docs/`：用户侧与实现侧说明层

## 还没确认的点

1. TUI 是否在运行时依赖 app-server 作为中间层，还是只在特定模式下使用？
2. `core` 和 `exec` 的关系：`exec` 是薄入口还是另一条重要主线？
3. `state` crate 与 `rollout/session` 的真实分工边界还要继续确认。
