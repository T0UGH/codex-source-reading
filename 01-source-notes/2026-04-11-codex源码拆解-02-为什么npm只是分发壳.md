# Codex 源码拆解 02：为什么 npm 只是分发壳

## 这篇看什么

回答一个很容易误判的问题：**Codex 明明是 Rust 项目，为什么用户主安装方式却是 npm？**

## 先给结论

npm 在 Codex 里扮演的是 **distribution layer（分发层）**，不是 runtime 主体。

更准确地说：

```text
npm / brew / release
    ↓
launcher / installer
    ↓
native Rust binary
    ↓
Codex runtime
```

所以：
- npm 负责用户触达和安装体验
- 真正运行的还是 Rust 编出来的原生二进制

## 关键代码路径

- `README.md`
- `codex-cli/package.json`
- `codex-cli/bin/codex.js`
- `docs/install.md`

## 关键证据

### 1. README 的安装说明
`README.md` 主推：
- `npm install -g @openai/codex`
- `brew install --cask codex`

这说明 OpenAI 把 npm / brew 作为面向开发者的主分发渠道。

### 2. npm 包的 bin 暴露
`codex-cli/package.json`：
- 包名：`@openai/codex`
- `bin.codex = bin/codex.js`

说明 npm 包本身只暴露一个 Node 入口脚本。

### 3. Node 入口脚本的真实行为
`codex-cli/bin/codex.js` 做的事不是实现业务逻辑，而是：
- 判断当前平台和架构
- 选择对应 target triple
- 定位平台对应的原生包 / vendor binary
- 最后 `spawn(binaryPath, process.argv.slice(2))`

这说明 JS 只是 **launcher**。

## 设计判断

### 1. 这是务实的产品分发决策
目标不是“纯 Rust 分发最优雅”，而是“让开发者最低摩擦安装 Codex”。

对于目标用户：
- Node / npm 常见
- Rust toolchain 不一定有
- cargo 构建依赖过重

所以用 npm 做分发壳是合理选择。

### 2. Rust 和 npm 在这里不是竞争关系，而是分层协作
- Rust：核心 runtime、性能、系统能力、跨平台原生执行
- npm：安装入口、版本分发、开发者生态触达

### 3. 这也说明不要把 `codex-cli/` 当实现主体
如果后续阅读把注意力放在 npm 层，会被带偏。

## 当前边界判断

- `codex-cli/` 不负责核心 agent 逻辑
- `codex-cli/bin/codex.js` 负责平台选择 + 原生 binary 启动
- 真正主体在 `codex-rs/`

## 还没确认的点

1. 各平台二进制包的具体构建链条还没继续往下追
2. brew 分发和 npm 分发是否共用完全相同的原生 artifact 还没确认
3. vendor/path 注入机制是否还承担额外依赖注入职责，可后续细看
